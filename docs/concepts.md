# Concepts

This page explains the five core primitives in Treco and how they relate to each other.

---

## Workspace

A workspace is the top-level tenant boundary. Every ticket, agent, and event belongs to exactly one workspace. All database queries filter by `workspace_id` — data from one workspace is never visible to another.

When you run `treco init`, you choose a workspace ID. The ID is a plain string you pick (default: `demo`). For teams, use a shared workspace ID so all agents report to the same dashboard.

---

## Ticket

A ticket represents a unit of work — a GitHub issue, Jira task, Linear issue, or a manually created task.

### Sources

Tickets can come from:

| Source | How to import |
|--------|---------------|
| GitHub Issues | `treco import <issue-url>` |
| Linear | `treco import <issue-url>` |
| Jira | `treco import <issue-url>` |
| Asana | `treco import <issue-url>` |
| Manual | `treco new "<title>"` |

### Body immutability

When a ticket is imported, the raw provider JSON is stored verbatim in the `body` field and never modified. This is an invariant — no code path mutates `body` after creation. The raw payload is your source of truth for what the ticket looked like at import time.

### Normalized fields

`title`, `description`, and `acceptance_criteria` are derived from `body` at import time by a source-specific adapter. They are never set from arbitrary user input after creation. If a provider field is missing, the adapter returns `None` rather than fabricating a value.

### Ticket status

| Status | Meaning |
|--------|---------|
| `open` | Not yet started |
| `in_progress` | An agent has emitted `ticket_started` |
| `done` | Agent emitted `done` event |
| `error` | Agent emitted `error` event |

---

## Agent

An agent is a registered process that reports work against tickets. Agents authenticate via an SDK key (`X-Agent-Key` header). The raw key is returned once on creation and never stored — only its SHA-256 hash lives in the database.

### Agent status

| Status | Meaning |
|--------|---------|
| `idle` | No active ticket session |
| `working` | Actively processing a ticket |
| `awaiting_approval` | Paused, waiting for human review |
| `error` | Last session ended with an error event |

An agent holds a `current_ticket_id` while working. When a session ends (`done` or `error` event), the agent returns to `idle` and `current_ticket_id` is cleared.

### API keys

API keys are generated by Treco on agent creation. The key is shown to the user once. Internally, only `api_key_hash` (SHA-256) is persisted. Keys are never logged, never included in error responses, and never appear in any API response after the creation call.

---

## Event

Events are the append-only record of everything an agent did. No event is ever updated or deleted. The event stream is the foundation for cost accounting, criteria tracking, and audit history.

### Event types

| Type | Emitted when |
|------|-------------|
| `ticket_started` | Agent begins a ticket session |
| `criterion_checked` | Agent marks one acceptance criterion complete |
| `criterion_failed` | Agent marks one criterion as failed |
| `pr_opened` | Agent opens a pull request |
| `log` | Agent emits a free-form message |
| `heartbeat` | Periodic keep-alive from a running agent |
| `deviation` | Agent departed from the expected plan |
| `done` | Agent successfully completes the ticket |
| `error` | Agent session ended with an error |

### Token accounting

Each event carries `tokens_in`, `tokens_out`, and `model`. These fields default to zero — not every event represents an LLM call. Cost is computed at read time by summing token counts across all events for a ticket. No derived cost value is ever persisted.

### Payload

The `payload` field is a free-form JSON object. Content depends on event type: a `log` event puts the message in `payload.message`, a `pr_opened` event puts the URL in `payload.url`, an `error` event puts details in `payload.error`. The schema is not enforced beyond being valid JSON.

---

## Acceptance Criteria

Acceptance criteria are the conditions that must be true for a ticket to be considered done. They are extracted from the ticket description at import time and stored as a list of criterion objects on the ticket.

### Extraction

Treco tries two strategies in order:

1. **LLM extraction** — sends the ticket description to `claude-haiku-4-5-20251001` (or `gpt-4o-mini`) with a prompt to return a JSON list of criteria. Used when `ANTHROPIC_API_KEY` or `OPENAI_API_KEY` is set.
2. **Checkbox fallback** — parses Markdown checkboxes (`- [ ] text`) from the description directly. Used when no LLM key is configured or when LLM extraction fails.

If both strategies fail, the ticket is created with an empty criteria list. Ticket creation is never blocked by extraction failure.

### Criterion structure

Each criterion is a JSON object:

```json
{
  "id": "3f1ab0a1-318c-423f-88a7-a7c88703f93a",
  "text": "Retry on 5xx up to 3 times",
  "done": false
}
```

`id` is a stable UUID. Use it when calling `treco check <id>` or when emitting `criterion_checked` events via the SDK.

### Marking criteria done

From the CLI:

```bash
treco check 3f1ab0a1-318c-423f-88a7-a7c88703f93a
```

From the SDK:

```python
await client.check(ticket_id, criterion_id="3f1ab0a1...", tokens_in=800, tokens_out=220)
```

Both paths emit a `criterion_checked` event. The criterion's `done` field is set to `true` on the ticket. A `criterion_failed` event sets it back to `false`.

## How They Compose

```
Workspace
  └── Ticket (one per issue/task)
        ├── acceptance_criteria (extracted at import)
        └── AgentEvents (append-only stream)
              ├── ticket_started
              ├── log / heartbeat / deviation
              ├── criterion_checked  ──▶  criteria[n].done = true
              ├── pr_opened
              └── done  ──▶  ticket.status = done

Agent
  └── authenticates via X-Agent-Key (stored hashed)
  └── owns the events it emits (agent_id on each event)
```

A single ticket can have events from multiple agents if work is handed off. An agent can only have one `current_ticket_id` at a time — to switch tickets, the current session must end first.
