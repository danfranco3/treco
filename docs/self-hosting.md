# Self-Hosting Treco

Run Treco on your own infrastructure with Docker and PostgreSQL. The default local setup uses SQLite; this guide covers production deployment.

---

## Prerequisites

- Docker 24+ and Docker Compose v2
- A terminal with port 8001 and 3000 available

---

## Docker Compose — Quickstart

The repo ships a `docker-compose.yml` that starts the backend, frontend, and a PostgreSQL 16 database together.

```bash
git clone https://github.com/your-org/treco
cd treco

# Create backend env file (minimum required vars)
cat > backend/.env <<EOF
JWT_SECRET=$(openssl rand -hex 32)
EOF

docker compose up --build
```

- Backend: `http://localhost:8001`
- Frontend: `http://localhost:3000`
- PostgreSQL: `localhost:5432` (user `treco`, password `treco`, db `treco`)

The backend waits for Postgres to pass its healthcheck before starting, then runs `alembic upgrade head` automatically.

## Environment Variables

Set these in `backend/.env` (for Docker Compose) or as environment variables in your hosting platform.

### Required

| Variable | Description |
|----------|-------------|
| `JWT_SECRET` | Random secret for signing JWTs. Must be at least 32 bytes. Never reuse across environments. |

Generate a secure value:

```bash
openssl rand -hex 32
```

### Database

| Variable | Default | Description |
|----------|---------|-------------|
| `DATABASE_URL` | `sqlite+aiosqlite:///./treco.db` | SQLAlchemy async connection string. |
| `DATABASE_MODE` | `sqlite` | Set to `postgres` when using PostgreSQL. |

PostgreSQL connection string format:

```
postgresql+asyncpg://USER:PASSWORD@HOST:PORT/DBNAME
```

The `+asyncpg` driver is required — `psycopg2` and synchronous drivers will not work with the async engine.

### LLM (for criteria extraction)

At least one is needed to extract acceptance criteria from ticket descriptions. If neither is set, Treco falls back to parsing markdown checkboxes.

| Variable | Description |
|----------|-------------|
| `ANTHROPIC_API_KEY` | Used when `LLM_PROVIDER=anthropic` (default). |
| `OPENAI_API_KEY` | Used when `LLM_PROVIDER=openai`. |
| `LLM_PROVIDER` | `anthropic` or `openai`. Default: `anthropic`. |

### GitHub OAuth (optional)

Required only if you want users to log in with GitHub.

| Variable | Description |
|----------|-------------|
| `GITHUB_CLIENT_ID` | From your GitHub OAuth app settings. |
| `GITHUB_CLIENT_SECRET` | Keep this secret — never commit or log it. |
| `FRONTEND_URL` | Base URL of your frontend. Used to build the OAuth redirect URI. Default: `http://localhost:3000`. |
| `BACKEND_URL` | Base URL of your backend. Used by spawned agent subprocesses. Default: `http://localhost:8001`. |

### CORS

| Variable | Default | Description |
|----------|---------|-------------|
| `CORS_ORIGINS` | `["http://localhost:3000", "http://localhost:8001"]` | JSON list of allowed origins. Set to your frontend URL in production. |

Example:

```bash
CORS_ORIGINS=["https://treco.yourdomain.com"]
```

## Frontend Environment Variables

The frontend build bakes in these values at build time:

| Variable | Description |
|----------|-------------|
| `NEXT_PUBLIC_API_URL` | URL of the backend. E.g. `https://api.yourdomain.com`. |
| `NEXT_PUBLIC_WORKSPACE_ID` | Default workspace ID shown in the dashboard. Default: `default`. |

Pass these as Docker build args:

```bash
docker build \
  --build-arg NEXT_PUBLIC_API_URL=https://api.yourdomain.com \
  --build-arg NEXT_PUBLIC_WORKSPACE_ID=default \
  ./frontend
```

Or set them in `docker-compose.yml` under `frontend.build.args`.

## Database Migrations

Migrations run automatically when the container starts (`alembic upgrade head` in `entrypoint.sh`). No manual step needed on first deploy or after upgrades.

To run migrations manually against a remote database:

```bash
cd backend
DATABASE_URL=postgresql+asyncpg://user:pass@host/db alembic upgrade head
```

## Deploy to Render

The repo includes `render.yaml` for one-click deployment to [Render](https://render.com).

1. Fork the repo.
2. In the Render dashboard, create a new **Blueprint** and point it at your fork.
3. Render reads `render.yaml` and provisions backend, frontend, and a free PostgreSQL instance.
4. Set `JWT_SECRET`, `ANTHROPIC_API_KEY`, and `CORS_ORIGINS` in the Render environment editor.

The `JWT_SECRET` is auto-generated by Render (`generateValue: true`) — copy it from the dashboard if you need it elsewhere.

## Production Checklist

Before going live:

- **`JWT_SECRET`** — set to a random 32+ byte value. The default `dev-secret-change-in-production` must not be used in production.
- **`CORS_ORIGINS`** — restrict to your actual frontend domain. Wildcard origins allow any site to call your API.
- **HTTPS** — terminate TLS in front of both backend and frontend. The OAuth callback (`/auth/github/callback`) requires HTTPS in production.
- **GitHub OAuth app** — set the callback URL in your OAuth app settings to `https://api.yourdomain.com/auth/github/callback`. Mismatched URLs cause silent auth failures.
- **Database** — use a managed PostgreSQL instance (Render, RDS, Supabase) with automated backups. SQLite is not recommended for multi-user production use.
- **Secrets** — never commit `backend/.env`. It is gitignored. Rotate `JWT_SECRET` and `GITHUB_CLIENT_SECRET` if either is ever exposed.
